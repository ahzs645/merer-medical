# Architecture: local-first, Convex-shaped

Mere Medical is a browser-first personal health record. Data lives on the
client; the only optional server component is `apps/api` (NestJS), which
handles OAuth callbacks for portals like Epic and Cerner. The web app can
run with no backend at all when data ingest happens via `.emrpkg` import
instead of portal sync.

## Two stores, one direction of travel

Today there are two client-side data stores in the codebase:

```text
                 ┌──────────────────────────────────────────┐
                 │              apps/web (React)             │
                 │  ─────────────────────────────────────    │
                 │  Repositories (UserRepository,            │
                 │  ConnectionRepository, ClinicalDocs)      │
                 │  Providers, services/fhir/*, etc.         │
                 └────────────┬──────────────────────┬──────┘
                              │                      │
                              ▼                      ▼
                    ┌──────────────────┐   ┌────────────────────┐
                    │   RxDB (today)   │   │   Dexie (future)   │
                    │  user_documents  │   │  users             │
                    │  connections     │   │  connections       │
                    │  clinical_docs   │   │  clinical_documents│
                    │  vector_storage  │   │  attachments       │
                    │  ...             │   │  ...               │
                    └──────────────────┘   └────────────────────┘
```

- **RxDB** is the original store. Schema lives in `apps/web/src/models/*`,
  collections are wired in `apps/web/src/app/providers/RxDbProvider.tsx`.
- **Dexie** is the new store, defined in `packages/local-dexie/src/db.ts`,
  intentionally shaped like the future Convex schema (camelCase, document
  IDs generated client-side, indexes that mirror Convex `defineTable`).

The leaf repositories (`User`, `Connection`, `ClinicalDocument`) now branch
on `localStorage.getItem('mere.useDexieRepos') === 'true'`. When off
(default), RxDB is the source of truth. When on, Dexie is the source of
truth for *anything that goes through the repositories*; everything else
(FHIR portal services, sync provider, vectors, summary, timeline) still
talks to RxDB directly. See [migration.md](./migration.md) for the full
status.

## `packages/` layout

```text
packages/
  domain/         # @mere/domain   - Entity types, zod schemas, ID helpers.
                  #                  No runtime dependency on any store.
  data/           # @mere/data     - AppDataClient interface, factory, React
                  #                  hooks. UI talks to this, never to Dexie
                  #                  or Convex directly.
  local-dexie/    # @mere/local-dexie
                  #                - Dexie schema + DexieDataClient
                  #                  implementing AppDataClient. Also exports
                  #                  the .emrpkg pack/unpack helpers.
```

Path mappings are declared once in `tsconfig.base.json`, so `apps/web` and
the packages all resolve `@mere/*` via the existing webpack
`tsconfig-paths` plugin.

## The `AppDataClient` interface

`packages/data/src/AppDataClient.ts` defines the verbs the UI uses:

```ts
interface AppDataClient {
  mode: 'local' | 'convex';
  users: UserCommands;
  userPreferences: UserPreferencesCommands;
  connections: ConnectionCommands;
  clinicalDocuments: ClinicalDocumentCommands;
  attachments: AttachmentCommands;
  instanceConfig: InstanceConfigCommands;
  summaryPagePreferences: SummaryPagePreferencesCommands;
  package: PackageCommands;
  close(): Promise<void>;
}
```

Each `*Commands` group is task-oriented (`users.create`, `users.select`,
`connections.bySource`) rather than CRUD-shaped. This avoids leaking
storage details into call sites and keeps Convex mutations one-to-one
with the methods here.

Two adapters implement the interface:

| Adapter             | Lives in                                  | Status      |
| ------------------- | ----------------------------------------- | ----------- |
| `DexieDataClient`   | `packages/local-dexie/src/dexieDataClient.ts` | Implemented |
| `ConvexDataClient`  | (future) `packages/convex-adapter/`           | Not started |

`createDataClient({ mode: 'local' })` dynamic-imports `@mere/local-dexie`
so a bundle that only ever uses one mode doesn't pull in the other.

## App-level IDs, plain JSON records

Two conventions keep the future Convex migration cheap:

1. **App-level IDs.** Every record carries `id: string` generated by
   `createId()` from `@mere/domain`. When we move to Convex, that becomes
   the `appId` field — Convex's internal `_id` is never seen by app code.
2. **Plain JSON-shaped records.** Timestamps are `number` (epoch ms),
   never `Date`. No `Map`/`Set`/class instances in stored values.
   Documents stay under Convex's 1 MB / nesting limits.

The legacy RxDB schemas don't meet these conventions (snake_case, no
generated IDs in some collections). The bridge in
`apps/web/src/repositories/dexie-bridge/translators.ts` does the mapping
when a request flows through the migrated repositories.

## The bridge (legacy ↔ new)

```text
       UI / hooks / services
              │
              ▼
   apps/web/src/repositories/*  ───────┐
              │                         │
   isDexieReposEnabled() ?              │
       │                │               │
       ▼                ▼               │
     RxDB           dexie-bridge        │
                          │             │
                          ▼             │
              translators (snake↔camel) │
                          │             │
                          ▼             │
                 @mere/data AppDataClient
                          │
                          ▼
                  DexieDataClient
                          │
                          ▼
                 packages/local-dexie/db
```

The bridge also provides `wrapAsRxDocument(...)` — a `Proxy` that fakes
RxDB's `.toJSON()`, `.toMutableJSON()`, `.update({ $set })` and
`.remove()` methods, plus field-level property access — so existing
consumers that hold `RxDocument<T>` handles keep working unchanged.

## Future state

```text
Phase 1 (today partial)
  Browser + IndexedDB only
  Dexie under the AppDataClient interface
  .emrpkg as the only ingest/egress if you skip apps/api

Phase 2
  Same UI, Convex adapter implementing AppDataClient
  Accounts/auth via Convex
  Realtime via useQuery

Phase 3
  Local-first + cloud sync
  Either via Convex's replication primitives, or by treating .emrpkg as a
  one-time migration step into Convex
```

The point of the design is that going Phase 1 → Phase 2 should replace
**the adapter**, not the app.
